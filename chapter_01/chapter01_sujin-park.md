## 1장 HTTP 개관

HTTP 완벽 가이드 책을 읽고 이해한 내용을 정리한 글입니다.

---

이 장은 HTTP 에 대해 다음 항목들에 대해 개략적으로 알려줍니다.

- 얼마나 많은 클라이언트가 서버와 통신하는가
- 리소스가 어디서 오는지
- 웹 트랜잭션이 어떻게 동작하는지
- HTTP 통신을 위해 사용하는 메시지의 형식
- HTTP 기저의 TCP 네트워크 전송
- 여러 종류의 HTTP 프로토콜
- 인터넷 곳곳에 설치된 다양한 HTTP 구성요소

### HTTP란?

---

```HTTP``` 는 이미지, HTML, 텍스트 파일, 동영상, 음성파일 등 웹 서버로부터 대량의 정보를 빠르고, 간편하고, 정확하게 웹 브라우저로 옮겨줍니다. HTTP 는 데이터 전송 프로토콜을 사용하기 때문에, 사용자는 HTTP 를 통해 받은 파일이 전송 중 파괴되거나, 중복되거나, 왜곡되는 것을 걱정하지 않아도 됩니다. 이는 또한 개발자에게 애플리케이션 고유의 기능을 구현하는 데에 집중할 수 있게끔 해줍니다.

### 웹 클라이언트와 서버

---

웹 콘텐츠는 웹 서버에 존재합니다. 웹 서버 (이하 HTTP 서버) 는 HTTP 클라이언트가 요청한 데이터를 저장하고, 요청한 데이터를 제공합니다. 

우리가 흔히 아는 HTTP 클라이언트는 인터넷 익스플로러나 구글 크롬같은 웹 브라우저입니다. 웹 브라우저는 서버에게 HTTP 객체를 요청하고 사용자의 화면에 보여줍니다. [http://github.com/sujin-park/deep-dive-dev](http://github.com/sujin-park/deep-dive-dev) 링크로 접근하면 웹브라우저는 HTTP 요청을 [http://github.com/sujin-park/deep-dive-dev](http://github.com/sujin-park/deep-dive-dev) 서버로 보냅니다. 서버는 요청받은 객체를 찾고, 성공했다면 요청받은 객체의 타입, 길이 등의 정보와 함께 HTTP 응답을 클라이언트에게 보냅니다.

### 리소스

---

웹 서버는 웹 리소스를 관리하고 제공합니다. 웹 리소스는 HTML, 워드 파일, 이미지, 동영상 파일 등 정적 파일이 있는데 반드시 정적 파일일 필요는 없습니다. 동적 리소스는 사용자가 누구인지, 어떤 정보를 몇 시에 요청했는지에 따라 다른 콘텐츠를 생성합니다.

웹 서버는 모든 HTTP 객체 데이터에 MIME 타입을 붙입니다. 브라우저는 서버로부터 객체를 돌려받을 때, 다룰 수 있는 객체인지 MIME 타입을 통해 확인 할 수 있습니다. 대부분의 웹브라우저는 수백 가지의 객체 타입을 다룰 수 있습니다.

MIME 타입은 **주 타입 / 부타입** 으로 이루어진 문자열 라벨입니다.

- text/html
- text/plain
- image/jpeg
- image/gif

웹 서버 리소스는 각자 이름을 갖고 있기 때문에, 클라이언트는 관심 있는 리소스를 지목할 수 있습니다. 서버 리소스 이름은 통합 자원 식별자, URI 로 불립니다. 정보 리소스를 고유하게 식별하고 위치를 지정할 수 있습니다.

URI 에는 통합 자원 지시자(uniform resource locator, URL)와 유니폼 리소스 이름(uniform resource name, URN) 이 있습니다.

- URI
    - 특정 서버의 한 리소스에 대한 구체적인 위치를 서술
    - 스킴 (scheme): 리소스에 접근하기 위해 사용되는 프로토콜 (ex. http://)
    - 서버의 인터넷 주소: ex) [github.com](http://github.com)
    - 웹 서버의 리소스: /sujin-park/deep-dive-dev/README.md

- URN
    - 콘텐츠를 이루는 리소스에 대해, 리소스의 위치에 영향 받지 않는 유일무이한 이름 역할을 합니다.
    - URN 은 인터넷 표준 문서 'RFC 2141' 가 어디에 있거나 상관없이 그것을 지칭하기 위해 사용할 수 있습니다. **urn:ietf:rfc:2141**

### HTTP 트랜잭션

---

HTTP 트랜잭션은 요청 명령, 응답 결과로 구성되어 있습니다.

**메서드**

HTTP 메서드는 여러가지 종류의 요청 명령을 지원합니다. 모든 HTTP 요청 메시지는 한개의 메서드를 가집니다.

- GET : 서버에서 클라이언트로 지정한 리소스를 보내라
- PUT : 클라이언트에서 서버로 보낸 데이터를 지정한 이름의 리소스로 저장하라
- DELETE : 지정한 리소스를 서버에서 삭제하라
- POST : 클라이언트 데이터를 서버 게이트웨이 애플리케이션으로 보내라
- HEAD : 지정한 리소스에 대한 응답에서, HTTP 헤더 부분만 보내라

**상태 코드**

모든 HTTP 응답 메시지는 상태 코드와 함께 반환합니다.

- 200 : 문서가 바르게 반환되었습니다.
- 302 : 요청한 리소스가 일시적으로 이동 되었음을 가리킵니다.

브라우저는 사용자를 이 URL의 페이지로 리다이렉트시키지만 검색 엔진은 그 리소스가 일시적으로 이동되었다고 해서 그에 대한 링크를 갱신하지는 않는다 ('SEO 관점' 에서 말하자면, 링크 주스(Link Juice)가 새로운 URL로 보내지지는 않는다).

침고: [https://developer.mozilla.org/ko/docs/Web/HTTP/Status/302](https://developer.mozilla.org/ko/docs/Web/HTTP/Status/302)

- 404 : 해당 리소스를 찾을 수 없습니다.

HTTP 는 상태 코드에 텍스트로 된 reason phrase 도 함께 보냅니다.

- 200 OK
- 200 Document attached
- 200 Success

**메시지**

웹 클라이언트에서 웹 서버로 보낸 HTTP 메시지를 **요청 메시지,** 서버에서 클라이언트로 가는 메시지는 **응답 메시지**라고 합니다.

시작줄

메시지의 첫 줄로, 요청이라면 무엇을 해야 하는지 응답이라면 무슨 일이 일어났는지 나타냅니다.

헤더

시작줄 다음에는 헤더 필드가 나오고 **key: value 로** 구성되며 헤더는 빈 줄로 끝납니다. 

본문

요청의 본문은 웹 서버로 데이터를 보내며, 응답의 본문은 클라이언트로 데이터를 반환합니다.

### TCP 커넥션
---
**TCP/IP**

HTTP 는 애플리케이션 계층 프로토콜입니다. HTTP는 네트워크 통신의 핵심적인 세부사항에 대해 신경 쓰지 않고 인터넷 전송 프로토콜인 TCP/IP 에게 맡깁니다.

TCP 는 다음을 제공합니다.

- 오류 없는 데이터 전송
- 순서에 맞는 전달 (데이터는 언제나 보낸 순서대로 도착한다)
- 조각나지 않는 데이터 스트림 (언제든 어떤 크기로든 보낼 수 있다)

TCP/IP 는 TCP 와 IP 가 층을 이루는, 패킷 교환 네트워크 프로토콜의 집합입니다. TCP/IP 는 각 네트워크와 하드웨어의 특성을 숨기고, 어떤 종류의 컴퓨터나 신뢰성 있는 의사소통을 하게 해줍니다.

TCP 커넥션이 맺어지면 클라이언트와 서버 간에 교환되는 메시지가 없어지거나 순서가 뒤바뀌어 수신되는 일은 없습니다.

네트워크 개념상, HTTP 프로토콜은 TCP 위의 계층입니다. HTTP 는 자신의 메시지 데이터를 전송하기 위해 TCP 를 사용합니다.

**접속, IP 주소, 포트번호**

HTTP 클라이언트가 서버에 메시지를 전송하기 전에, IP 주소와 포트번호를 사용해 클라이언트와 서버 사이에 TCP/IP 커넥션을 맺어야 합니다.

TCP 는 필요한 정보인 서버 컴퓨터에 대한 IP 주소와 포트번호를 URL에서 얻을 수 있습니다.

[http://127.0.0.1](http://127.0.0.1):8080/index.html 에서 IP 는 127.0.0.1 (로컬), 포트번호는 8080 입니다.

숫자로 된 IP 주소가 없는 URL 이 대부분인데 이들은 IP 주소에 대한 이해하기 쉬운 형태의 별명인 **호스트명(도메인)을  사용합니다.**

호스트 명은 DNS (도메인 이름 서비스) 를 통해 쉽게 IP 로 변환될 수 있습니다.

URL 에 포트번호가 없다면 기본 값은 80번 포트입니다.

아래는 웹 브라우저가 HTTP를 이용해서 서버의 HTML 리소스를 사용자에게 보여주는 과정입니다.

1. 브라우저는 서버의 URL에서 호스트 명을 추출합니다.
2. 브라우저는 서버의 호스트 명을 IP로 변환합니다.
3. 브라우저는 URL에서 포트번호를 추출합니다.
4. 브라우저는 서버와 TCP 커넥션을 맺습니다.
5. 브라우저는 서버에 HTTP 요청을 보냅니다.
6. 서버는 브라우저에 HTTP 응답을 돌려줍니다.
7. 커넥션이 닫히면, 브라우저는 문서를 보여줍니다.

### 프로토콜 버전

- HTTP/0.9

    오직 GET 메서드만 지원하고, 멀티미디어 콘텐츠에 대한 MIME 타입이나 HTTP 헤더, 버전 번호는 지원하지 않습니다. HTTP/0.9 는 단순히 HTML 객체를 받아오기 위해 만들어진 것입니다.

- HTTP/1.0

    버전 번호, HTTP 헤더, 추가 메서드, 멀티미디어 객체 처리를 추가하였습니다. 시각적으로 매력적인 웹페이지와 상호작용하는 폼을 실현했고, 이는 www 를 대세로 만들었습니다.

- HTTP/1.0+

    오래 지속되는 keep-alive 커넥션, 가상 호스팅 지원, 프락시 연결 지원을 포함해 많은 기능이 추가되었습니다.

- HTTP/1.1

    HTTP 설계의 구조적 결함 교정, 두드러진 성능 최적화, 잘못된 기능 제거에 집중하였습니다.

- HTTP/2.0

    HTTP/1.1 성능 문제를 개선하기 위해 구글의 SPDY 프로토콜 기반으로 설계가 진행 중인 프로토콜입니다.


### 웹의 구성요소

- 프락시

    프락시는 클라이언트와 서버 사이에 위치하여 클라이언트의 모든 HTTP 요청을 받아 서버에 전달합니다. 프락시는 주로 보안을 위해 사용되며 요청과 응답을 필터링해줍니다.

- 캐시

    웹캐시와 캐시 프락시는 HTTP 프락시 서버로 클라이언트가 같은 문서를 요청하면 캐시가 갖고 있는 사본을 바로 받아서 훨씬 빠르게 문서를 받을 수 있습니다.

    캐시를 효율적으로 동작하게 하고 캐시된 콘텐츠를 최신 버전으로 유지하면서 동시에 프라이버시도 보호하기 위한 기능을 정의합니다.

- 게이트웨이

    게이트웨이는 주로 HTTP 트래픽을 다른 프로토콜로 변환하기 위해 사용됩니다. 진짜 서버인 것처럼 요청을 다루기 때문에 클라이언트는 자신이 게이트웨이와 통신하고 있음을 알아채지 못합니다.

- 터널

    터널은 두 커넥션 사이에서 raw 데이터를 열어보지 않고 그대로 전달해줍니다. HTTP 터널은 주로 비 HTTP 데이터를 하나 이상의 HTTP 연결을 통해 그대로 전송해주기 위해 사용됩니다.

    대표적인 예로, 암호화된 SSL 트래픽을 HTTP 커넥션으로 전송함으로써 웹 트래픽만 허용하는 사내 방화벽을 통과시키는 것이 있습니다.

    우선 HTTP/SSL 터널은 HTTP 요청을 받아들여 목적지의 주소와 포트번호로 커넥션을 맺고 이후부터는 암호화된 SSL 트래픽을 HTTP 채널을 통해 목적지 서버로 전송할 수 있게 됩니다.

- 에이전트

    HTTP 요청을 만들어주는 프로그램입니다. 예를 들어 사람의 통제 없이 스스로 웹을 돌아다니며 HTTP 트랜잭션을 일으키고 콘텐츠를 받아오기도 합니다.
